## 1. Authentication flow

**Model:** **Not OAuth2 for the HermX↔SnapTrade plane.** SnapTrade uses a **proprietary HMAC request-signature scheme** ("Request Signatures") on every API call. The official SDKs generate the signature automatically; a raw-HTTP client must implement it. The end-user's *brokerage* login is OAuth/broker-native, but that happens **inside SnapTrade's connection portal** — HermX never touches it.

**Two credential layers (matches TD §6.1 planes 2 & 3, with one correction — see §4):**

| Credential | Scope | Notes |
|---|---|---|
| `clientId` | Partner (HermX) | Public-ish identifier. |
| `consumerKey` | Partner (HermX) | **Sensitive**; signs every request AND verifies webhooks (§4). Redact. |
| `userId` | Per end-user | Immutable, caller-chosen. Docs: **do not use email**. |
| `userSecret` | Per end-user | Randomly generated by SnapTrade at registration; **sensitive** — store encrypted. |

**Flow (exact endpoints):**

1. **Register user** — `POST /snapTrade/registerUser`
   Body: `{ "userId": "<caller-chosen>" }` → Response: `{ "userId", "userSecret" }`.
2. **Generate connection portal URL** — `POST /snapTrade/login?userId=<>&userSecret=<>`
   Body (all optional): `broker`, `immediateRedirect`, `customRedirect`, `reconnect`, `connectionType`, `connectionPortalVersion`, `showCloseButton`, `darkMode`.
   Response: **`{ "redirectURI", "sessionId" }`**. The user opens `redirectURI`, authenticates **directly with their brokerage** inside SnapTrade's portal. `customRedirect` is the return URL (the "loginRedirectUri" equivalent).
3. **Re-auth / fix a broken connection** — same `POST /snapTrade/login` with **`reconnect`** set (pass the connection/authorization id). This is how a `CONNECTION_BROKEN` account is re-linked without losing the mapping.
4. **Rotate the user secret** — `POST` `Authentication_resetSnapTradeUserSecret` (rotate a compromised `userSecret`).
5. **Delete user** — `Authentication_deleteSnapTradeUser`. **List users** — `GET` list-users.

**Tokens returned:** no bearer/access token. The durable secrets are `userSecret` (per user) and `consumerKey` (partner). Every subsequent call is authenticated by signing with `consumerKey` and passing `userId`/`userSecret` as query params.

> **Answers TD §10 Q-implicit (auth model):** proprietary HMAC signing, not OAuth2; `redirectURI` is the portal link; `reconnect` handles re-auth.

## 2. Account / listing endpoints

**List connected accounts:** `GET /accounts` → array of account objects (each has an `id` = accountId UUID, `brokerage_authorization`, `institution_name`, `name`, `number`, `total_value`, etc.).

**Positions / holdings:** `GET /accounts/{accountId}/positions`
> The bare `/positions` path is marked **deprecated** in favor of a newer unified holdings endpoint; the field shape below is stable across both.

Position object fields:

| Field | Type | Meaning |
|---|---|---|
| `symbol` | object | nested; `symbol.symbol.symbol` (string) is the ticker; `symbol.symbol.description`, `.currency`, `.exchange`, `.type` |
| `units` | number \| null | share/contract quantity (long-only retail ⇒ positive) |
| `price` | number \| null | last/market price |
| `open_pnl` | number \| null | **unrealized P&L** |
| `average_purchase_price` | number \| null | cost basis |
| `currency` | object | position currency |
| `cash_equivalent` | boolean \| null | money-market/cash-like flag |
| `tax_lots` | array | per-lot detail |

**Balances:** `GET /accounts/{accountId}/balances`

| Field | Type | Meaning |
|---|---|---|
| `currency` | object | `currency.id` (uuid), `currency.code` (ISO-4217, e.g. `"USD"`), `currency.name` |
| `cash` | number \| null | available cash; **can be negative in margin accounts**; includes money-market funds |
| `buying_power` | number \| null | **margin-specific**; docs: "for non-margin accounts, buying power should be the same as cash"; may be absent on some brokers |

> **Answers TD §10 Q9 (buying-power semantics):** the **conservative `avail` field is `cash`**, not `buying_power`. `buying_power` includes margin and may be `null`; `cash` is the safe over-sizing guard.

**Orders:** `GET /accounts/{accountId}/orders` — array of order objects (shape identical to the place-order response, §3). **Recent orders** is a separate lighter endpoint (`getUserAccountRecentOrders`).

**Order status enum (authoritative — for the normalizer state map):**
`NONE, PENDING, ACCEPTED, FAILED, REJECTED, CANCELED, PARTIAL_CANCELED, CANCEL_PENDING, EXECUTED, PARTIAL, REPLACE_PENDING, REPLACED, EXPIRED, QUEUED, TRIGGERED, ACTIVATED`.
Docs: "SnapTrade does a best effort to map brokerage statuses to statuses in this enum."

**Activity / transactions feed:** account activities endpoint returns the richer feed (trades, dividends, transfers, fees, corporate actions) — maps to HermX `get_order_history_raw`.

## 3. Trading endpoints ⭐ (the critical section)

### One-step vs two-step — BOTH exist; one-step is recommended

- **One-step (recommended):** `POST /trade/place` — "Place equity order" (a.k.a. force order). Places directly; broker may reject.
- **Two-step:** `POST /trade/impact` (check) → returns a `trade.id` (tradeId) → `POST /trade/{tradeId}` (place checked). The **tradeId expires after 5 minutes.**

> **Answers TD §10 Q2 (one-step vs two-step):** SnapTrade offers both. HermX should use the **one-step `POST /trade/place`** for the normal path (fewer hops, lower ambiguity, no 5-min expiry race). Reserve `/trade/impact` for `plan()`/dry-run preview (it returns estimated commission, forex fees, and post-trade `remaining_cash`).

### ⭐ Idempotency — YES, natively supported

`POST /trade/place` accepts a request-body field:

> **`client_order_id`** — type `string (uuid) | null`, description (verbatim): *"Optional caller-supplied identifier passed through to the brokerage for idempotent order placement. Must be a canonical 36-character UUID."*

> **Answers TD §10 Q1 (idempotency key) — the hard blocker for live: RESOLVED = YES.** HermX passes a UUID-form key derived from `cl_ord_id` as `client_order_id`; SnapTrade passes it through to the brokerage for source-level idempotent placement. Constraint: it **must be a canonical 36-char UUID** — HermX `cl_ord_id` values that are not UUIDs must be deterministically mapped to a UUID (e.g. UUIDv5 of the `cl_ord_id`). This does **not** eliminate the HermX-side dedupe + write-ahead journal + no-blind-retry rule; it *adds* a source-level backstop. Note the two-step `/trade/{tradeId}` path relies on tradeId (single-use, 5-min) for its idempotency instead.

### `POST /trade/place` request body

| Field | Type | Req | Notes |
|---|---|---|---|
| `account_id` | string(uuid) | ✅ | target account |
| `action` | string | ✅ | `BUY, SELL, BUY_TO_OPEN, BUY_TO_CLOSE, SELL_TO_OPEN, SELL_TO_CLOSE` |
| `universal_symbol_id` | string(uuid) \| null | ⚠️ | one of this or `symbol` required |
| `symbol` | string \| null | ⚠️ | ticker alternative to `universal_symbol_id` |
| `order_type` | string | ✅ | `Market, Limit, Stop, StopLimit` |
| `time_in_force` | string | ✅ | `Day, GTC, FOK, IOC, GTD` |
| `units` | number \| null | ⚠️ | one of `units` or `notional_value` (fractional allowed) |
| `notional_value` | number \| null | ⚠️ | dollar-based alternative to `units` |
| `price` | number \| null | ⚠️ | required for `Limit`/`StopLimit` |
| `stop` | number \| null | ⚠️ | required for `Stop`/`StopLimit` |
| `trading_session` | string | ❌ | e.g. extended hours |
| `expiry_date` | string(date-time) \| null | ❌ | for `GTD` |
| **`client_order_id`** | string(uuid) \| null | ❌ | **idempotency key (above)** |

### `POST /trade/place` response (and `GET .../orders` / `POST /trade/{tradeId}` share this order shape)

`brokerage_order_id` (string), `status` (enum §2), `universal_symbol` \| `option_symbol` (object\|null), `action` (string), `total_quantity` / `open_quantity` / `filled_quantity` / `canceled_quantity` (string\|null), `execution_price` / `limit_price` / `stop_price` (decimal-string\|null), `order_type` (string\|null), `time_in_force` (string), `time_placed` (date-time), `time_executed` (date-time\|null), `time_updated` (date-time\|null).

> Note: quantities and prices come back as **strings** (decimal-as-string), not floats — the normalizer must parse them.

### Cancel

`POST /accounts/{accountId}/orders/cancel`, body `{ "brokerage_order_id": "<string>" }`.
> Marked deprecated in favor of a newer `Trading_cancelOrder`; the `brokerage_order_id`-in-body contract is what's documented today.

### Symbol lookup (to obtain `universal_symbol_id`)

`GET` `ReferenceData_getSymbols` (search) or `ReferenceData_symbolSearchUserAccount` (account-scoped). Response `id` = the `universal_symbol_id`. Quotes: `GET /trading/quotes` (also yields `universalSymbolId`). HermX can skip the lookup by passing raw `symbol` on `/trade/place`, at the cost of broker-side symbol resolution.

### Other asset classes

`POST` place **option** order (single/multi-leg) and place **crypto** order exist (distinct bodies; crypto uses `instrument.symbol`/`instrument.type`, `amount`, `post_only`). **Options remain deferred** per TD non-goals (multi-leg shape).

## 4. Webhook events

**Delivery model:** **push** — SnapTrade POSTs JSON to a URL you configure in the **Dashboard → Webhooks tab** (not per-API-call). No polling required.

**Event types (exact names):**
`USER_REGISTERED, USER_DELETED, CONNECTION_ATTEMPTED, CONNECTION_ADDED, CONNECTION_DELETED, CONNECTION_BROKEN, CONNECTION_FIXED, CONNECTION_UPDATED, CONNECTION_FAILED, NEW_ACCOUNT_AVAILABLE, ACCOUNT_TRANSACTIONS_INITIAL_UPDATE, ACCOUNT_TRANSACTIONS_UPDATED, ACCOUNT_REMOVED, ACCOUNT_HOLDINGS_UPDATED, TRADE_DETECTION, TRADE_UPDATE`.

HermX-relevant mapping:
- **`TRADE_DETECTION` / `TRADE_UPDATE`** → order-fill / status-change → reconciliation nudge.
- **`CONNECTION_BROKEN` / `CONNECTION_FAILED`** → `ConnectionStore.mark_unhealthy` → fail-closed auth-health gate. **`CONNECTION_FIXED`** → mark healthy.
- **`ACCOUNT_HOLDINGS_UPDATED`** → invalidate aggregation cache.

**Payload fields (common):** `webhookId`, `clientId`, `eventTimestamp` (ISO-8601), `userId`, `eventType`, plus event-specific (`brokerageId`, `accountId`, `connectionAttemptedResult`, …). The old `webhookSecret` payload field is **deprecated** — do not rely on it.

**Verification (IMPORTANT CORRECTION to TD §6.1 / §2.6):**
The `Signature` header is an **HMAC-SHA256 of the raw request body keyed with your `consumerKey`**, base64-encoded. There is **no separate webhook secret** — the same `consumerKey` that signs API requests verifies webhooks.
⇒ **A dedicated `SNAPTRADE_WEBHOOK_SECRET` env var is unnecessary.** Webhook verification reuses `SNAPTRADE_CONSUMER_KEY`. (Auth-plane separation from the TradingView HMAC still holds — different key, different endpoint, different algorithm input — but the third plane is not an *independent secret*, it's the consumer key.)

**Delivery guarantees:** **at-least-once.** If the handler does not return `200/201/202/204`, SnapTrade retries **up to 3 times starting at ~30-minute intervals with exponential backoff**. **No ordering guarantee** across events. ⇒ handler must be **idempotent** (rely on the journal `can_transition` guard) and treat webhooks as an optimization over the poll backstop.

> **Answers TD §10 Q4 (events + delivery + signature): RESOLVED** — events enumerated, at-least-once, no ordering, HMAC-SHA256/consumerKey/base64.

## 5. Rate limits

- **Global:** **250 requests/minute** per client, rolling window, across all endpoints/accounts.
- **Headers:** `X-RateLimit-Limit`, `X-RateLimit-Remaining`, `X-RateLimit-Reset` (**seconds until window reset**).
- **Throttle status:** **HTTP 429**. **No `Retry-After` header** — compute wait from `X-RateLimit-Reset`.
- **Per-account tier (Personal keys only):** an **additional 10 req/min per account** on ~9 data endpoints (holdings, balances, positions, orders, recent orders, order detail, quotes, …). **Commercial keys are exempt** from this per-account tier.
- **Backoff guidance (from docs):** honor `X-RateLimit-Reset`; exponential backoff + **jitter**; distribute reads across accounts rather than hammering one.

> **Answers TD §10 Q6 (rate limits): RESOLVED** — 250/min global; 10/min/account on data endpoints for Personal keys. This directly sizes the aggregation poll cadence: **for a Personal key, per-account reads must stay ≤10/min**, so batch positions+balances+orders per account and cache aggressively.

## 6. Error codes (⚠️ partial — see note)

The dedicated `/docs/errors` page returned **404** at the guessed URL, so this section is grounded only in what other pages state. **Flagged as an open item to confirm from the live reference before coding `errors.py`'s full map.**

Confirmed:
- **429** — rate limit (§5); retryable after `X-RateLimit-Reset`.
- **Order rejection** — a placed order can be **rejected by the brokerage** (invalid order / insufficient funds); surfaces as order `status` = `REJECTED`/`FAILED`, i.e. a **terminal REJECTED**, not a transport error. This is the "explicit reject → REJECTED" branch of TD §7.1.
- Standard HTTP semantics apply for 400 (bad request), 401/403 (bad signature / unauthorized), 404 (unknown user/account/order).
- SnapTrade error responses carry a structured body (`code` + human `detail`/`description`) — **exact numeric SnapTrade error-code catalogue not verified here.**

**Retryable vs non-retryable (HermX mapping, per TD §7.1):**
- Retryable (reads): 429, 5xx, network timeout → bounded backoff.
- **Writes:** timeout/5xx/ambiguity → **UNKNOWN, never blind-retry** (now backed by `client_order_id` idempotency if a retry is ever attempted).
- Non-retryable terminal: brokerage `REJECTED`/`FAILED` status, 400 validation.

> **TD §10 Q-implicit (error taxonomy): PARTIALLY RESOLVED** — enough to map modes; confirm the numeric error catalogue from the live errors page in Phase 0.

## 8. Sandbox

**A true dedicated sandbox exists** — but with a decisive limitation for HermX.

- **Availability:** enabled **by default on non-production keys** (personal keys + commercial *test* keys). No request needed.
- **What it simulates:** a **proprietary simulated brokerage** — full connection flow, accounts, balances, holdings, orders, transactions. You pick a **scenario** at connect time:
  - Data: **Self-directed** (default, funded), **Cash only**, **No transactions**, **No accounts**.
  - Error: **Invalid credentials**, **Account locked**, **Rate limited**.
- **Transactions covered:** trades, cash transfers, dividends, corporate actions, options events, multi-account.
- **⚠️ KEY LIMITATION:** the sandbox is **READ-ONLY — placing/canceling trades is NOT supported.** Data is **static/simulated** with relative timestamps.

> **Answers TD §10 Q8 (true sandbox vs Alpaca-paper-only): RESOLVED with a fork:**
> - **Phase 1 (aggregation / read path):** the **SnapTrade sandbox is the ideal test bed** — it exercises connection flow, accounts, positions, balances, orders, transactions, *and* error scenarios (broken-connection, rate-limited) **with zero real linking**. Golden fixtures for the normalizer can come straight from sandbox.
> - **Phase 2 (order placement / write path):** the sandbox **cannot** test `place`/`cancel`. Order-placement testing **must** use a **real Alpaca Paper connection** (a live SnapTrade connection to a paper brokerage). So "demo" for a SnapTrade *write* strategy = **Alpaca Paper**, not the SnapTrade sandbox.
